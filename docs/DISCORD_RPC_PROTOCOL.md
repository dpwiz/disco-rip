# Discord RPC Protocol Documentation

## Introduction

Discord RPC (Rich Presence) is a local Inter-Process Communication (IPC) protocol that allows external applications to communicate with the running desktop instance of Discord. It is most commonly used to update a user's "Rich Presence" profile (displaying detailed status information, such as the game they are currently playing or the file they are editing), but the underlying RPC protocol can theoretically support other commands as well.

The connection between the client application and Discord is made locally on the user's machine. The application must establish a connection, perform a handshake, and then it can exchange commands and events with Discord via JSON payloads wrapped in binary frames.

## System Primitives and Connection

Depending on the operating system, the IPC connection is established using different system primitives.

### Windows
On Windows, the protocol uses **Named Pipes**.
The client should attempt to connect to:
`\\?\pipe\discord-ipc-0`

If `discord-ipc-0` is not found or fails, it should iterate from `0` to `9` (e.g., `\\?\pipe\discord-ipc-1`, `\\?\pipe\discord-ipc-2`, etc.) to find the active Discord pipe.

### Linux and macOS
On Unix-based systems, the protocol uses **Unix Domain Sockets**.
The socket file is typically named `discord-ipc-0` (up to `discord-ipc-9`, similar to Windows).

The client should look for the socket in the following directories, in order of preference:
1. `$XDG_RUNTIME_DIR`
2. `$TMPDIR`
3. `$TMP`
4. `/tmp`

Example paths: `/run/user/1000/discord-ipc-0` or `/tmp/discord-ipc-0`.

## Transport Protocol (The Frame Layer)

Once connected to the pipe or socket, all communication is done using a simple framed binary protocol. Each message sent and received consists of an 8-byte header followed by a JSON string payload.

### Frame Structure

| Offset | Type                        | Description |
|--------|-----------------------------|-------------|
| `0x00` | 32-bit Integer (Little-Endian) | Opcode      |
| `0x04` | 32-bit Integer (Little-Endian) | Length of the JSON payload |
| `0x08` | Variable string             | JSON payload (UTF-8 encoded) |

### Opcodes

The opcode defines the type of the frame. The known opcodes are:
- `0` - **Handshake**: Used only for the initial connection payload.
- `1` - **Frame**: Used for all standard JSON payloads (commands, events, responses).
- `2` - **Close**: Instructs the other side to close the connection, or notifies of a close.
- `3` - **Ping**: Used to keep the connection alive.
- `4` - **Pong**: Response to a Ping.

## Application Protocol (The Message Layer)

After the connection is established at the transport layer, the client and Discord communicate using JSON payloads.

### 1. Handshake

The very first message sent by the client must be a Handshake frame (Opcode `0`). The payload must contain the version of the protocol and the Application Client ID (obtained from the Discord Developer Portal).

**Client -> Discord (Handshake Payload)**
```json
{
  "v": 1,
  "client_id": "123456789012345678"
}
```

If successful, Discord will respond with a `Frame` (Opcode `1`) containing an introductory `DISPATCH` event, signaling the `READY` state.

**Discord -> Client (Ready Response)**
```json
{
  "cmd": "DISPATCH",
  "evt": "READY",
  "data": {
    "v": 1,
    "config": {
      "cdn_host": "cdn.discordapp.com",
      "api_endpoint": "//discord.com/api",
      "environment": "production"
    },
    "user": {
      "id": "123456789",
      "username": "User",
      "discriminator": "1234",
      "avatar": "abcdef123456",
      "bot": false,
      "flags": 0,
      "premium_type": 1
    }
  }
}
```

### 2. General RPC Commands

After the handshake, all messages from the client use Opcode `1` (Frame).
The general shape of an RPC request from the client is:

```json
{
  "cmd": "COMMAND_NAME",
  "args": {
    ...
  },
  "nonce": "unique-request-id-12345"
}
```
- `cmd`: The command being executed (e.g., `SET_ACTIVITY`, `SUBSCRIBE`, `AUTHORIZE`).
- `args`: The arguments corresponding to the command.
- `nonce`: A unique identifier (typically a UUID) generated by the client to match the response from Discord to the specific request.

Discord will respond with Opcode `1` and a JSON payload mirroring the `nonce`:

```json
{
  "cmd": "COMMAND_NAME",
  "data": {
    ...
  },
  "evt": null,
  "nonce": "unique-request-id-12345"
}
```

### 3. Setting Rich Presence (`SET_ACTIVITY`)

The most common use case is updating the user's Rich Presence. This uses the `SET_ACTIVITY` command.

**Client -> Discord**
```json
{
  "cmd": "SET_ACTIVITY",
  "args": {
    "pid": 12345,
    "activity": {
      "state": "In a Multiplayer Match",
      "details": "Playing on Map X",
      "timestamps": {
        "start": 1610000000,
        "end": 1610005000
      },
      "assets": {
        "large_image": "image_key_large",
        "large_text": "Large Image Tooltip",
        "small_image": "image_key_small",
        "small_text": "Small Image Tooltip"
      },
      "party": {
        "id": "party-1234",
        "size": [2, 4]
      },
      "secrets": {
        "join": "secret-join-key",
        "spectate": "secret-spectate-key",
        "match": "secret-match-key"
      },
      "instance": true,
      "buttons": [
        {
          "label": "Play Now!",
          "url": "https://example.com/play"
        }
      ]
    }
  },
  "nonce": "a-unique-nonce-string"
}
```

*Note on `pid`: The OS process ID is usually passed. Discord monitors this process and will clear the presence if the process terminates.*

**Discord -> Client (Success Response)**
```json
{
  "cmd": "SET_ACTIVITY",
  "data": {
    "state": "In a Multiplayer Match",
    "details": "Playing on Map X",
    ...
  },
  "evt": null,
  "nonce": "a-unique-nonce-string"
}
```

## Summary

1. **Find IPC primitive**: Locate the Named Pipe or Unix Domain Socket `discord-ipc-0`.
2. **Connect**: Establish a stream connection.
3. **Handshake**: Send Opcode `0` with the `client_id`. Wait for `READY`.
4. **Send Commands**: Send Opcode `1` containing `SET_ACTIVITY` or other commands as JSON payloads.
5. **Listen**: Read incoming 8-byte headers to determine payload length, then read the JSON payload for responses or dispatch events.
